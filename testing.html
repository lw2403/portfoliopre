A/B Testing

Plant Shop

Overview

I designed 2 versions of a plant shop website, and performed an A/B test on the two designs The goal is to measure which of the two designs is better at achieving a predefined task and goal. The task for users is to checkout their cart with at least $150 of cacti/succulents. The 2 metrics that were run on statistical tests were: Time to Completion and Return Rate.

Timeline:2 weeks

Tools:Python + Heroku Logs + Visual Studio

Context:CS1300 Responsive Redesign

Context

This is one of the projects from CS1300( an UIUX class at Brown). I first designed the tested websites with different design choices and formed statistical hypothesis. Then, I asked my friends to do the task to help me collect the data. I then extracted data from the Heroku logs, and transitioned the data to an excel sheet. Next, I cleaned and ran the data with scripts I developed in an earlier assignment. Lastly, I analyzed and interpreted the staistiscal results and analyzed it in further details. I find this project to be a particularly interesting project because I learned to use data and statistical analysis to gain deeper insights into the effectiveness of my own design choices.

T-TEST

A test for Time to Completion

Time to Completion

I ran the T-Test from the python scripts I developed in an earlier class assignment to analyze time for a user to complete his or her $150 task. I defined the test, and explained my reasons for my alternative hypothesis below.

Definition of Time to Completion:

The time difference of when a user starts and finishes the $150 task. 

Null Hypothesis:

The true difference between time to completion for design A and design B is zero.

Alternative Hypothesis:

The true difference between time to completion for design A and design B is different from zero. 

A’s Layout

In design A, all items are on the left. Each row has 1 item. This will make users have to scroll down more and do more work to see all the items. Therefore, I think this layout will require more time for completion. And it’s an inefficient use of space.

A’s Shopping Cart Button

Shopping Cart is on the bottom left side of the page. It’s less obvious to users and they may need more time to find a shopping cart checkout button because usually it’s located at the bottom right. Also, more scrolling is required to reach the bottom of the page.

B’s Shopping Cart Button

Design B’s Shopping Cart is in a more optimal spot because the distance between the items and the button is much closer than A. Users don’t need to scroll to the bottom in order to checkout items.

B’s Information Display

Design B’s Shopping Cart is in a more optimal spot because the distance between the items and the button is much closer than A. Users don’t need to scroll to the bottom in order to checkout items.

Result

i failed to reject the null hypothesis. From the data output , we can see 95% of the time, the true difference will be 0. My p-value of 0.0567 is slightly larger than 0.05. So, I fail to reject the null hypothesis of no difference between my design A and design B time to completion. However I think that there is likely a difference between the 2 designs, but the sample is not big or diverse enough that we still fail to reject the null hypothesis.

CHI-SQUARED TEST

A test for Return Rate

Return Rate

I ran the Chi-squared Test from the python scripts I developed in an earlier class assignment to analyze time for a user to complete his or her $150 task. I defined the test, and explained my reasons for my alternative hypothesis below.

Definition of Return Rate:

Users are counted as “returned” users when they return to the shopping page after checkout. 

Null Hypothesis:

The true difference between return rate for design A and design B is zero.

Alternative Hypothesis:

The true difference between return rate for design A and design B is different from zero. 

A’s Information Display

In design A, “Add To Cart” button is below the item description and price. This design is more readable for users to get information on what they want to buy. Users see the description and price first which helps them do mental math to add things up to $150 so they don’t have to return to the page.

B’s Information Display

Version B’s “Add To Cart” button is above the item description and price. Users see a button to add before they see description and price. This can make users not add things up to $150 in the first go because they are likely to just click add, and return to the page if they don’t meet $150.

A’s Button & Layout

Shopping Cart is at a less optimal spot because it’s on the bottom left side of the page. There is only 1 item per line, users need to scroll more. So, they don’t want to come back to this page and do everything again. Therefore, they rather achieve $150 in one go, and not return to the page after clicking checkout.

B’s Button & Layout

B’s layout lets users see all the items and the shopping cart button on one page. No scrolling is required. Less scrolling needed, and users will want to avoid doing mental math. They expect to not meet $150 in the first go, and rather return to the page to add more items.

Result

i failed to reject the null hypothesis. From the data output , we can see 95% of the time, the true difference will be 0. My p-value of 0.06188 is slightly larger than 0.05. So, I fail to reject the null hypothesis of no difference between design A and design B return rate. However this t-score is big, and therefore I think that there is likely a difference between the 2 design. However the restriction including sample size is not diverse enough that we still fail to reject the null hypothesis.

CONCLUSION

What I learned about the project and A/B testing

Results

I learned that my designs for interface A and interface B were not significant to Time to Completion & Return Rate think the result may have been related to reasons I listed in the previous section such as the sample is not large and diverse enough. However, through comparing individual user sessions, I see that users mostly had a shorter time to completion and lower return rate for design B which is what I initially thought.

Take-away

In general, this is one of my favourite projects in the class because I was taught to use data to gain insights into my design. Overall, I realized a more optimal layout, navigation, and affordability can save users’ time to meet their goal even though my project results were not significant. I look forward to use A/B testing in the future on my other projects.
